# Middleware

## Execution model

`process` is called **after** the reducer has already applied the action to state — `state` always reflects the updated value. All registered middleware run **concurrently** via `withTaskGroup`; there is no sequential chain.

Calling `dispatch` schedules a brand-new action through the full Redux cycle (reducer → all middleware). It does not pass control to another middleware. Calling `dispatch` with the same action that triggered `process` is a no-op — it is silently filtered to prevent trivial loops.

```
Action → Reducer (sync) → new state → all middleware in parallel (async)
                                        ↓ each may call dispatch(newAction)
                                        Action → Reducer → ...
```

## Writing a middleware

Conform to `MiddlewareType` as a `struct`. Use the `DispatchClosure<Action>` typealias for the parameter type.

```swift
struct SearchMiddleware: MiddlewareType, Sendable {
    let api: APIClient

    func process(
        action: SearchAction,
        state: SearchState,
        dispatch: @escaping DispatchClosure<SearchAction>
    ) async {
        guard case .queryChanged(let query) = action else { return }
        let results = (try? await api.search(query)) ?? []
        await dispatch(.resultsLoaded(results))
    }
}
```

Use `struct` (not `class`) — all stored properties are `Sendable`, so the compiler verifies thread safety automatically without needing `@unchecked Sendable`.

If your middleware never dispatches a follow-up action (e.g. analytics, logging), simply omit the `dispatch` call:

```swift
struct LoggerMiddleware<Action: Actionable, State: Statable>: MiddlewareType, Sendable {
    func process(action: Action, state: State, dispatch: @escaping DispatchClosure<Action>) async {
#if DEBUG
        print("[Store] ▶ \(action)")
#endif
    }
}
```

## Debouncing with `CancellableTask`

Conform to `CancellableTask` to get per-key task cancellation. Calling `run(key:)` with the same key cancels any in-flight task registered under that key before starting the new one.

This is the standard pattern for search-as-you-type, delayed autosaves, and any other "only the last one matters" scenario.

```swift
struct SearchMiddleware: MiddlewareType, CancellableTask, Sendable {
    let taskManager = TaskCancellationManager()
    let api: APIClient

    func process(
        action: SearchAction,
        state: SearchState,
        dispatch: @escaping DispatchClosure<SearchAction>
    ) async {
        guard case .queryChanged(let query) = action else { return }

        await run(key: "search") {
            // Debounce: if another queryChanged arrives within 300ms, this task
            // is cancelled — Task.sleep throws CancellationError and we return early.
            do {
                try await Task.sleep(for: .milliseconds(300))
            } catch {
                return
            }

            let results = (try? await self.api.search(query)) ?? []
            await dispatch(.resultsLoaded(results))
        }
    }
}
```

### How it works

- `CancellableTask` requires one stored property: `taskManager: TaskCancellationManager`.
- `TaskCancellationManager` is an actor — safe to share across any isolation context.
- `run(key:)` cancels the previous task registered under `key` and starts a new one immediately.
- Finished tasks are removed from the manager automatically, keeping memory bounded.

### Multiple independent keys

Each key is independent. You can debounce different operations in the same middleware without them interfering:

```swift
await run(key: "search") { ... }
await run(key: "autocomplete") { ... }
```

### Manual cancellation

```swift
await cancel(key: "search")   // cancel one key
await cancelAll()              // cancel everything
```

## Type-erasing middleware manually

The `@MiddlewareResultBuilder` attribute (generated by `@StoreView`) handles type erasure automatically. If you compose a store without the macro, wrap each middleware in `AnyMiddleware`:

```swift
StoreContainerView(reducer: SearchReducer()) {
    AnyMiddleware(SearchMiddleware(api: api))
} content: { store in
    ...
}
```

Or use the `@MiddlewareBuilder` result builder directly:

```swift
StoreContainerView(reducer: SearchReducer()) {
    SearchMiddleware(api: api)   // type-erased automatically by the builder
} content: { store in
    ...
}
```

## Awaitable dispatch — `.refreshable` and async contexts

`dispatch` is fire-and-forget: it applies the reducer synchronously and schedules middleware in an unstructured `Task`. The caller returns immediately — the middleware is still running.

This is fine for most interactions, but SwiftUI's `.refreshable` modifier expects an `async` closure that only returns when the work is done. If you use plain `dispatch`, the spinner dismisses the moment `dispatch` returns, before any network call completes.

Use `dispatchAsync` instead. It applies the reducer synchronously, then **suspends until all first-level middleware complete**:

```swift
// ❌ Spinner dismisses immediately — middleware still running
.refreshable {
    store.dispatch(.refresh)
}

// ✅ Spinner stays visible until middleware finishes
.refreshable {
    await store.dispatchAsync(.refresh)
}
```

`dispatchAsync` is available on `ObservableStore`, `ScopedStore`, and `Storable`. It behaves identically to `dispatch` in every other respect — the action goes through the reducer and all middleware, `onStateChange` fires, and the resulting state is observable.

### Cancellation

`dispatchAsync` tasks are tracked in `inflightTasks` alongside regular `dispatch` tasks, so `cancel()` and store deallocation cancel them correctly. If the caller's `Task` is cancelled (e.g. the view is dismissed mid-refresh), the middleware task is also cancelled cooperatively via `withTaskCancellationHandler`.

### ScopedStore

`dispatchAsync` is available on `ScopedStore` as well, mapping the child action to the parent before suspending:

```swift
struct FeedView: View {
    let store: ScopedStore<HomeReducer, FeedState, FeedAction>

    var body: some View {
        List { ... }
            .refreshable {
                await store.dispatchAsync(.refresh)
            }
    }
}
```

## Cycle detection

The store ships two built-in cycle detectors. Both are diagnostic-only — they never drop or delay a dispatch.

### Async cycle detection (default: enabled)

`maxActionFrequency` (default: 20) triggers a console warning when the same action is dispatched more than N times within `cycleWindow` (default: 1 second). This catches middleware loops like "A dispatches B, B dispatches A".

```
[ReduxCore] ⚠️ Possible async cycle: 'queryChanged("abc")' dispatched 21 times in < 1.0 seconds
```

Adjust the threshold if your feature legitimately dispatches an action frequently. Pass the argument directly to `StoreContainerView`, or via the `@StoreView` macro:

```swift
// Direct usage
StoreContainerView(
    reducer: SearchReducer(),
    maxActionFrequency: 100,
    cycleWindow: .seconds(2)
) { ... } content: { store in ... }

// Via macro
@StoreView(reducer: SearchReducer.self, maxActionFrequency: 100, cycleWindow: .seconds(2))
struct SearchScreen: View { ... }
```

Set `maxActionFrequency: 0` to disable async cycle detection entirely.

### Sync cycle detection

`maxDispatchDepth` limits how deeply one dispatch may synchronously trigger another. The default is `.max` (unlimited). Lower it during debugging to break infinite re-entrant chains:

```swift
// Direct usage
StoreContainerView(
    reducer: SearchReducer(),
    maxDispatchDepth: 10
) { ... } content: { store in ... }

// Via macro
@StoreView(reducer: SearchReducer.self, maxDispatchDepth: 10)
struct SearchScreen: View { ... }
```

When the limit is hit:

```
[ReduxCore] ⚠️ Max dispatch depth (10) exceeded. Breaking potential cycle for action: queryChanged("abc")
```

Both detectors are active only in `DEBUG` builds.
