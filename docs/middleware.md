# Advanced Middleware

## Debouncing with `CancellableTask`

Conform to `CancellableTask` to get per-key task cancellation. Calling `run(key:)` with the same key cancels any in-flight task registered under that key before starting the new one.

This is the standard pattern for search-as-you-type, delayed autosaves, and any other "only the last one matters" scenario.

```swift
struct SearchMiddleware: MiddlewareType, CancellableTask, Sendable {
    let taskManager = TaskCancellationManager()
    let api: APIClient

    func process(
        action: SearchAction,
        state: SearchState,
        dispatch: @escaping DispatchClosure<SearchAction>
    ) async {
        guard case .queryChanged(let query) = action else { return }

        await run(key: "search") {
            // Debounce: if another queryChanged arrives within 300ms, this task
            // is cancelled — Task.sleep throws CancellationError and we return early.
            do {
                try await Task.sleep(for: .milliseconds(300))
            } catch {
                return
            }

            let results = (try? await self.api.search(query)) ?? []
            await dispatch(.resultsLoaded(results))
        }
    }
}
```

### How it works

- `CancellableTask` requires one stored property: `taskManager: TaskCancellationManager`.
- `TaskCancellationManager` is an actor — safe to share across any isolation context.
- `run(key:)` cancels the previous task registered under `key` and starts a new one immediately.
- Finished tasks are removed from the manager automatically, keeping memory bounded.

### Multiple independent keys

Each key is independent. You can debounce different operations in the same middleware without them interfering:

```swift
await run(key: "search") { ... }
await run(key: "autocomplete") { ... }
```

### Manual cancellation

```swift
await cancel(key: "search")   // cancel one key
await cancelAll()              // cancel everything
```

## Type-erasing middleware manually

The `@MiddlewareResultBuilder` attribute (generated by `@StoreView`) handles type erasure automatically. If you compose a store without the macro, wrap each middleware in `AnyMiddleware`:

```swift
StoreContainerView(reducer: SearchReducer()) {
    AnyMiddleware(SearchMiddleware(api: api))
} content: { store in
    ...
}
```

Or use the `@MiddlewareBuilder` result builder directly:

```swift
StoreContainerView(reducer: SearchReducer()) {
    SearchMiddleware(api: api)   // type-erased automatically by the builder
} content: { store in
    ...
}
```

## Cycle detection

The store ships two built-in cycle detectors. Both are diagnostic-only — they never drop or delay a dispatch.

### Async cycle detection (default: enabled)

`maxActionFrequency` (default: 20) triggers a console warning when the same action is dispatched more than N times within `cycleWindow` (default: 1 second). This catches middleware loops like "A dispatches B, B dispatches A".

```
[ReduxCore] ⚠️ Possible async cycle: 'queryChanged("abc")' dispatched 21 times in < 1.0 seconds
```

Adjust the threshold if your feature legitimately dispatches an action frequently:

```swift
StoreContainerView(
    reducer: SearchReducer(),
    maxActionFrequency: 100,
    cycleWindow: .seconds(2)
) { ... } content: { store in ... }
```

Set `maxActionFrequency: 0` to disable async cycle detection entirely.

### Sync cycle detection

`maxDispatchDepth` limits how deeply one dispatch may synchronously trigger another. The default is `.max` (unlimited). Lower it during debugging to break infinite re-entrant chains:

```swift
StoreContainerView(
    reducer: SearchReducer(),
    maxDispatchDepth: 10
) { ... } content: { store in ... }
```

When the limit is hit:

```
[ReduxCore] ⚠️ Max dispatch depth (10) exceeded. Breaking potential cycle for action: queryChanged("abc")
```

Both detectors are active only in `DEBUG` builds.
